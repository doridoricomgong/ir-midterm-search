# 정보검색 중간고사 분석보고서

## 1. 시스템 구조

이번 실기 과제는 맨손으로 검색 프로그램 전체를 작성하는 것보다, 생성인공지능이 만든 프로그램이 수업에서 배운 정보검색 구조를 어떻게 구현했는지 분석하는 데 초점을 둔다. 구현한 시스템은 Python, NLTK, Streamlit을 사용한다. 말뭉치는 NLTK의 `Inaugural Address` 코퍼스이며, 중간고사 지시문 기준에 맞추어 조지 워싱턴부터 조 바이든까지의 취임사만 사용하도록 `2021-Biden.txt`까지만 필터링했다. 현재 NLTK 데이터에는 `2025-Trump.txt`도 포함되어 있으므로, 과제 조건과 맞추기 위해 Biden 이후 파일은 제외했다.

시스템은 크게 색인 모듈과 웹 인터페이스 모듈로 나뉜다. `ir_engine.py`는 문서 ID 부여, 대통령 이름과 연도 저장, 토큰화, stemming, 사전 구축, 역파일 구축, TF/IDF 계산, 평균 문서 길이 계산, cosine 랭킹, BM25 랭킹, 중요 문장 추출을 처리한다. `app.py`는 Streamlit UI로 검색창, 랭킹 방식 선택, 검색 결과, 질의어별 IDF, 검색된 취임사별 TF, 원문 보기 링크를 제공한다. 검증 URL은 로컬 실행 기준으로 `http://localhost:8501`이다.

## 2. AI 생성 코드의 처리 방식

첫 단계는 각 취임사에 문서 ID를 부여하는 것이다. NLTK fileid 순서대로 문서를 읽고, `1789-Washington.txt` 같은 파일명에서 연도와 대통령 이름을 분리한다. 이 과정은 강의에서 말한 문서 집합을 내부 docID 체계로 바꾸는 단계에 해당한다. 검색 결과에는 사람이 이해할 수 있도록 대통령 이름과 연도도 함께 표시한다.

두 번째 단계는 문서를 term 단위로 바꾸는 것이다. 생성된 코드는 정규식으로 영문 알파벳 토큰만 추출하고, 모두 소문자로 바꾼 뒤 `PorterStemmer`를 적용한다. 예를 들어 `Freedom`, `freedoms`는 모두 `freedom`으로 정규화된다. 이는 `lecture02-dictionary`에서 다룬 tokenization과 stemming 절차에 해당한다. 다만 stopword 제거는 하지 않았다. 과제 요구사항에는 stopword 제거가 없고, 취임사 규모가 작기 때문에 원래 term 분포를 유지하는 편이 TF/IDF 확인에 더 적합하다고 판단했다.

세 번째 단계는 사전과 역파일을 만드는 것이다. 각 문서의 token 목록을 `Counter`로 바꾸어 문서별 term frequency를 저장하고, term을 key로 하여 `{doc_id: tf}` 형태의 postings를 만든다. 이 구조는 `lecture01-intro`와 `lecture02-dictionary`에서 배운 dictionary와 postings list의 축소판이다. 대규모 시스템에서는 postings를 docID 순서로 정렬하고 압축하지만, 이 과제에서는 문서가 약 59개 수준이므로 메모리 dictionary로 충분하다.

네 번째 단계는 문서 빈도와 IDF 계산이다. 각 term이 몇 개의 문서에 등장하는지 document frequency를 구하고, `log((N+1)/(df+1))+1` 방식의 smoothing된 IDF를 계산한다. 이 값은 검색 화면의 “질의어 IDF” 표에 표시된다. 검색 결과마다 질의 term이 해당 취임사에 몇 번 등장했는지도 TF 표로 별도 표시한다. 이는 중간고사 지시문의 “질의문의 각 term에 대한 idf, 검색한 취임사마다 각 term에 대한 TF를 볼 수 있어야 함” 조건을 충족한다.

랭킹은 두 가지 방식으로 처리했다. Cosine 방식은 문서와 질의를 TF-IDF 벡터로 보고 정규화된 dot product를 계산한다. 이는 `lecture06-tfidf`와 `lecture07-vectorspace`의 vector space model과 직접 연결된다. BM25 방식은 `k1=1.2`, `b=0.9`를 고정하고, term frequency를 문서 길이와 평균 문서 길이로 보정한다. 긴 취임사가 무조건 유리하지 않게 하는 점에서 단순 TF-IDF보다 현실적인 랭킹 함수로 볼 수 있다.

중요 문장 추출은 각 검색 결과에서 query term이 많이 나오고 IDF가 높은 문장을 고르는 방식으로 구현했다. 각 문장을 다시 tokenize/stem한 뒤, query term의 문장 내 빈도와 IDF를 곱해 점수를 계산한다. 가장 높은 점수의 문장을 snippet으로 표시한다. 이는 `lecture08-evaluation`에서 말한 dynamic summary, 즉 검색 질의에 따라 달라지는 요약의 단순 구현으로 해석할 수 있다.

## 3. 수업 내용과 비교

`lecture01-intro`의 핵심은 inverted index, dictionary, postings, Boolean query processing이다. 이번 시스템은 Boolean AND/OR 처리 대신 랭킹 검색을 중심으로 만들었지만, 내부 자료구조는 같은 방향이다. term별 postings를 만들고, 검색 시 query term의 postings에 포함된 문서만 후보로 삼는다. 즉 exact Boolean retrieval에서 ranked retrieval로 확장한 형태다.

`lecture02-dictionary`의 tokenization, stemming, dictionary entries, postings 개념은 구현에 직접 반영되었다. 반면 skip pointer, positional index, phrase query는 구현하지 않았다. 과제 요구사항이 취임사 단위 랭킹과 중요 문장 추출에 집중되어 있고, 코퍼스 크기가 작기 때문에 빠른 postings merge 최적화는 실질적인 효과가 작다.

`lecture03-tolerant`의 wildcard, spelling correction, k-gram index, edit distance도 구현 범위 밖이다. 이 기능들은 사용자가 `pro*` 같은 질의나 오타 질의를 넣을 때 필요하다. 현재 UI는 기본 검색창과 cosine/BM25 비교가 목적이므로 tolerant retrieval은 생략했다. 보고서 관점에서는 “AI가 문제 요구사항만 충족하고, 강의의 고급 검색 연산은 추가하지 않았다”는 점이 중요한 분석 포인트다.

`lecture04-construction`의 index construction은 작은 버전으로 반영되었다. 강의에서는 blocked sort-based indexing, distributed indexing, dynamic indexing 같은 대규모 색인 구축을 다루지만, 이번 프로그램은 모든 문서를 메모리에 올려 한 번에 색인을 만든다. 말뭉치가 작기 때문에 이 방식은 단순하고 검증하기 쉽다. 그러나 웹 규모 검색엔진이라면 메모리 한계, 디스크 postings, 동적 업데이트 문제가 추가되어야 한다.

`lecture05-compression`의 dictionary compression과 postings compression은 구현하지 않았다. 취임사 말뭉치는 압축이 필요한 규모가 아니며, 과제 제출에서 확인해야 하는 것은 압축 효율보다 TF, IDF, 역파일, 랭킹 계산이 제대로 드러나는지다. 따라서 압축 생략은 결함이라기보다 과제 목적에 맞춘 단순화라고 볼 수 있다.

`lecture06-tfidf`와 `lecture07-vectorspace`는 이번 구현의 핵심이다. 문서를 term-document vector로 보고 TF-IDF 가중치를 부여한 뒤 cosine으로 순위를 정한다. 질의도 짧은 문서처럼 tokenize/stem하여 같은 vector space에 놓는다. BM25는 강의의 TF/IDF 아이디어를 더 실용적인 랭킹 함수로 확장한 방식이며, 문서 길이 보정을 통해 긴 문서 편향을 줄인다.

`lecture08-evaluation`과 연결되는 부분은 결과 설명과 수동 검증이다. 과제에서는 정답 relevance set이 제공되지 않으므로 precision, recall, MAP을 엄밀히 계산하기 어렵다. 대신 `freedom union`, `war peace`, `constitution people` 같은 질의를 넣고, 결과가 관련 대통령 취임사와 관련 문장을 잘 보여주는지 확인한다. 이 방식은 정량 평가보다는 기능 검증과 질적 평가에 가깝다.

## 4. 검증 결과

자동 테스트는 pytest로 작성했다. 테스트는 tokenization과 stemming, dictionary 생성, inverted index의 TF 저장, IDF 계산, cosine/BM25 랭킹, Biden까지의 코퍼스 필터링을 검증한다. 특히 `freedom union` 질의에서 `Washington` 문서가 더 높은 점수를 받는지 확인했고, 검색 결과가 snippet과 TF 정보를 포함하는지도 확인했다.

수동 검증은 Streamlit 앱에서 진행한다. `freedom union`을 입력하면 자유와 연합을 많이 언급한 취임사가 상위에 나타난다. `war peace`는 전쟁과 평화를 다룬 취임사가 후보로 올라오며, `constitution people`은 헌법과 국민을 언급한 문서가 검색된다. 같은 질의를 cosine과 BM25로 번갈아 실행하면 순위나 점수 차이를 볼 수 있다. 또한 검색 결과의 “취임사 보기” 버튼을 누르면 `?doc=<fileid>` 형식의 내부 URL로 원문을 확인할 수 있다.

## 5. 한계와 결론

이 프로그램은 과제 요구사항을 중심으로 만든 작은 검색 시스템이다. 대규모 검색엔진에서 필요한 압축, skip pointer, positional index, wildcard query, spelling correction, distributed indexing은 구현하지 않았다. 그러나 수업 시간에 배운 핵심 흐름인 tokenization, stemming, dictionary, postings, TF, IDF, vector space ranking, BM25 ranking, dynamic snippet은 모두 확인할 수 있다.

생성인공지능은 기본 구조와 코드를 빠르게 만들 수 있지만, 그 결과가 강의 개념과 정확히 대응되는지 확인하는 일은 별도로 필요하다. 특히 NLTK 데이터가 최신화되어 `2025-Trump.txt`까지 포함된 점처럼, 과제 지시문과 실제 라이브러리 상태가 다를 수 있다. 따라서 AI가 만든 프로그램을 그대로 제출하는 것보다, 데이터 범위, 랭킹 공식, 파라미터, 출력 항목을 사람이 검증하고 수업 내용과 비교해 설명하는 것이 이번 과제의 핵심이라고 판단한다.
